# @jode/claude-code — remote environment (plan 02)

Worker → Durable Object → Container stack that runs the Claude Desktop app
headless in a Cloudflare container and serves its **real renderer SPA** to the
browser, wired back to the headless Electron main process over a `/bridge`
WebSocket relay. Lifted from the `cloudflare-split` prototype; see
[`../../plans/june-2026/02-cloudflare-claude-code.md`](../../plans/june-2026/02-cloudflare-claude-code.md).

## Layout

```
apps/claude-code/
├── src/index.ts            # Worker — edge router, SPA injection, /api tunnel
├── src/generated-assets.ts # GENERATED by gen-assets.mjs (index.html + bridge-client + mainView)
├── bridge/bridge.cjs       # in-container WS RPC relay (runs in Electron main)
├── browser-bridge/         # bridge-client.js injected into the served SPA
├── scripts/
│   ├── build-payload.mjs   # rebuilds payload/ from /Applications/Claude.app (self-contained)
│   ├── gen-assets.mjs      # bundles SPA assets into src/generated-assets.ts
│   ├── entrypoint.sh       # container supervisor (health → Xvfb → Electron bridge)
│   ├── xstartup.sh         # launches headless Electron + bridge under the Xvfb display
│   ├── install-electron-linux.sh
│   └── health-server.mjs
├── Dockerfile              # container image (bakes payload/ + bridge/)
├── wrangler.toml
└── payload/linux-rehost/   # GENERATED, gitignored — the rehosted app bundle
```

## Build pipeline (the payload is regenerable, not committed)

The 300MB+ `payload/linux-rehost/` tree is **not** in git. It is rebuilt from the
installed macOS Claude Desktop app by `build-payload.mjs`, which extracts
`Claude.app/Contents/Resources/app.asar`, merges the unpacked native modules,
and applies the Linux shims (`@ant/claude-swift` stub, helper shims,
`bootstrap.cjs`, the shell-path-worker platform patch) so the bundle boots
headless under forced-Linux Electron in the container.

```bash
# 1. Rebuild the payload from the installed app (needs Claude.app on this Mac;
#    override location with CLAUDE_APP_PATH=/path/to/Claude.app)
npm run build:payload

# 2. Bundle the SPA assets into src/generated-assets.ts
npm run gen

# 3. Deploy (predeploy re-runs gen)
npm run deploy
# or a no-publish check:
npm run dry-run
```

From the repo root the same steps are `npm run cf:build-payload`, `cf:gen`,
`cf:deploy`, `cf:dry-run`.

## Authentication — Cloudflare Access (single email)

The prototype's hardcoded `split-bridge-key-v1` shared key is **gone as an auth
mechanism**. Security is now two layers:

1. **Cloudflare Access** fronts the Worker hostname (the IdP-backed front door,
   policy = one allowed email). Access injects a signed JWT on every request.
2. **The Worker verifies that JWT** (`@jode/auth` → `verifyAccessJwt`) on *every*
   request including the `/bridge` WebSocket upgrade — RS256 signature against the
   team's JWKS, `iss`, `aud`, `exp`, and `email` claim. Fails closed.

The Worker→container hop keeps a secret (`BRIDGE_KEY`) purely as defense-in-depth
on the already-private Durable Object binding. It is **never** hardcoded: it lives
as a Worker secret and is injected into the container at start via `envVars`.

### One-time setup

```bash
# 1. Create a self-hosted Access application over the Worker's hostname, with a
#    policy allowing exactly jacques@comet.com (one-time PIN or OAuth IdP).
#    Dashboard: Zero Trust → Access → Applications → Add. Copy the Application
#    Audience (AUD) tag.

# 2. Put the non-secret config in wrangler.toml [vars]:
#      ACCESS_TEAM_DOMAIN = "https://<team>.cloudflareaccess.com"
#      ACCESS_AUD         = "<the AUD tag>"
#      ALLOWED_EMAIL      = "jacques@comet.com"

# 3. Set the Worker→container secret:
wrangler secret put BRIDGE_KEY      # e.g. value from: openssl rand -hex 32

# 4. Deploy
npm run deploy
```

Any identity that is not the allowlisted email is rejected at the edge (401/403).
For local `wrangler dev`, copy `.dev.vars.example` → `.dev.vars`.
